## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| ip-api default fields omit `offset` and `query` (and other extended fields) | ip-api's `/json/` endpoint returns only status/country/countryCode/region/regionName/city/zip/lat/lon/timezone/isp/org/as/query by default. Requesting `offset` (tz), `currency`, `mobile`, `proxy`, `hosting` requires an explicit `?fields=` query param. Manifested twice this session: Plan 03-02 shipped without `?fields=` and clock stayed at `--:--` forever (fix c23bb5a); Plan 04-01 caught this proactively by carrying the anti-pattern into the plan text. | advisory | Any future ip-api integration must specify `?fields=<comma-separated>` in the URL. When adding a new field, `grep -q "?fields=" bootstrap.py` and add the field name explicitly. |
| Symmetric pattern reuse hides upstream defects | Plan 03-02 D-31 mirror ("interval selection reads state inline") was structurally correct BUT assumed the upstream `offset` field was always populated. Plans that piggyback on an API round-trip to gain a new field must explicitly verify field name against the actual response, not just the API docs. | advisory | For any plan that adds a new field to `bootstrap.fetch()`, add an explicit verification step: "confirm the field name appears in the actual response". Consider a REPL check as part of the plan's `<verify>` block. |
| Stale docs after a rename | `.planning/codebase/*.md` (STACK.md, STRUCTURE.md, ARCHITECTURE.md, CONVENTIONS.md, INTEGRATIONS.md, CONCERNS.md, TESTING.md) still reference `weather.py` / `wifi.py`. `CLAUDE.md` was partially patched but historical narrative sections still contain some `wifi.py` / `weather.py` prose. This is deliberate v1-close technical debt — not to be fixed inline. | advisory | Run `/gsd:map-codebase` at v2 start to regenerate all codebase/*.md and CLAUDE.md from the current on-disk source. |
