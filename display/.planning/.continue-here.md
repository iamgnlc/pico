## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| ip-api default fields omit `offset` (and other extended fields) | ip-api's `/json/` endpoint returns only status/country/countryCode/region/regionName/city/zip/lat/lon/timezone/isp/org/as/query by default. Requesting fields like `offset`, `currency`, `mobile`, `proxy`, `hosting` requires an explicit `?fields=` query param. Plan 03-02 shipped assuming `offset` was in the default response; result: `loc.get("offset")` returned None, persistence file never got written, Clock view stayed at `--:--` indefinitely until fix `c23bb5a`. | advisory | Any future ip-api integration must specify `?fields=<comma-separated>` in the URL. When adding a new field, `grep -q "?fields=" weather.py` and add the field name explicitly. |
| Symmetric pattern reuse hides upstream defects | The Plan 03-02 D-31 mirror ("interval selection reads state inline") was structurally correct BUT assumed the upstream `offset` field was always populated. The plan's threat model included T-03-02-02 (malicious ip-api response) but did NOT enumerate "field missing from default response". Symmetry doesn't guarantee upstream contracts. | advisory | For any plan that piggybacks on an existing API round-trip to gain a new field, add an explicit verification step: "confirm the field name appears in the actual response, not just the API docs". Consider a manual REPL check as part of the plan's `<verify>` block. |
