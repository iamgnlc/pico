# Quick 260720-x55: Fall back to cached data on transient fetch failure

**One-liner:** Introduce a fourth `_cache_status` value `"stale"` in `views/weather_view.py` so a transient fetch failure keeps the last-good weather icon + temperature on screen instead of flipping to "no data".

## Problem

On a transient fetch failure (WiFi ok, but `ip-api` or `open-meteo` returns nothing), `bootstrap.fetch()` returns `(ip, None, None, None, None, None)`. The old `set_data` flipped `_cache_status` to `"no_data"` unconditionally, which caused `render()` to hide the previously-cached icon + temperature and show "no data" until the next successful fetch (~60s later). User-visible symptom: "after a while the display shows no data, even if the API is available."

## Fix — two surgical edits to `views/weather_view.py`

### Edit 1 — `set_data` (branch when `temp is None`)

Replaced the fixed `"no_data"` assignment with a ternary that probes the cache:

```python
# Preserve last-good cache on transient fetch failure — flip to "stale"
# only if we have something to fall back on; "no_data" if the cache is cold.
if temp is None:
    _cache_status = "no_data" if _cached_temp is None else "stale"
    return
```

`_cached_temp is None` is the cold-cache signal (matches the initial module-level value). Cached values are NOT cleared on this path — the whole point of the fix is to keep them available.

### Edit 2 — `render` (final branch)

Turned the dangling `else:` into an explicit `elif _cache_status == "ok" or _cache_status == "stale":`. Body is byte-identical to the prior `else:` — same `icons.draw(...)`, same temperature centering, same degree-glyph ellipse. Both statuses render from the cached values.
