<objective>
Fix a stale-cache-fallback bug in `views/weather_view.py`: on a transient fetch failure (bootstrap returns `(ip, None, None, None, None, None)` from bootstrap.py:53), the view currently flips `_cache_status` to `"no_data"` and hides the previously-cached icon + temperature until the next successful fetch (~60s later). User-visible symptom: "after a while the display shows no data, even if the API is available."

Purpose: The cache is there specifically so transient network hiccups don't blank a perfectly usable view. Preserve the "no data" UX only for the true cold-cache case (first fetch after boot failed, nothing to fall back to).

Output: A modified `views/weather_view.py` introducing a fourth `_cache_status` value `"stale"` — last fetch failed but cached data is available — and rendering the cached icon + temperature for both `"ok"` and `"stale"`. No other file touched.

Scope discipline: `bootstrap.py`, `main.py`, `views/clock_view.py`, `views/system_view.py`, `sh1107.py`, `icons.py`, `text_render.py` MUST stay byte-identical. No audit of clock/system for the same pattern — deliberate follow-up decision.
</objective>
