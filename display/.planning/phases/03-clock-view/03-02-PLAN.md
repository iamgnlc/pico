## Phase Goal (extension)

**As a** person using the Pico OLED **in any location without editing config files**, **I want** the Clock view to automatically derive its timezone from geolocation and remember it across reboots, **so that** the device shows correct local time immediately after network comes up on the next boot, without waiting for a weather fetch.

<objective>
Deliver the redirected CLOCK-02 semantics: replace the hardcoded `TZ_OFFSET` config from Plan 03-01 with a fully automatic system. Timezone offset is sourced from ip-api's `offset` field (already returned by the existing geolocation round-trip inside `weather.current()` — piggybacked, no new API call). The offset is persisted to `tz_offset.txt` on the Pico's flash and loaded at module import so subsequent boots have a good offset before the first weather fetch even completes.

Behavior contract:
- Clock view shows `HH:MM` iff BOTH `_synced == True` (NTP succeeded this boot) AND `_cached_tz_offset is not None` (either loaded from file at boot OR received from a weather fetch this session).
- Any other combination → `--:--`.
- Flash-wear guard: file is written only when the fetched offset differs from the cached value. Common case (stationary device, no DST) writes the file exactly once (first successful fetch on first-ever boot) then never again for the device's lifetime, minus DST transitions.

Output: `weather.py` extended to 4-tuple; `weather_view.py` unpacks and forwards to `clock_view.set_tz_offset`; `clock_view.py` gains persistence + a new module state field + a state-gated `render` and `should_tick`; `main.py` loses the `TZ_OFFSET` config line; `.gitignore` gains `tz_offset.txt`.
</objective>

### ip-api `offset` field (piggybacked from existing call)

`http://ip-api.com/json/` returns JSON that includes:
- `offset`: int, seconds from UTC (DST-aware, e.g. -25200 for PDT, -28800 for PST)
- `timezone`: string, IANA name (e.g. "America/Los_Angeles") — NOT used (would need a TZ database on-device)

The existing `weather.current()` already parses this JSON and uses `lat`/`lon`. Adding `offset` extraction is a single `loc.get("offset")` line. Failure (missing key, `loc` malformed) is caught by the existing broad `except Exception:` and returns `None` for the offset field.

### Target `weather.py`
```python
import urequests


def current():
    try:
        r = urequests.get("http://ip-api.com/json/")
        loc = r.json()
        r.close()
        offset = loc.get("offset")
        url = ("https://api.open-meteo.com/v1/forecast"
               "?latitude={}&longitude={}"
               "&current=temperature_2m,weather_code,is_day").format(
            loc["lat"], loc["lon"])
        r = urequests.get(url)
        cur = r.json()["current"]
        r.close()
        return cur["temperature_2m"], cur["weather_code"], cur["is_day"], offset
    except Exception:
        return None, None, None, None
```
