<objective>
Replace `main.py`'s `while True: weather_view.render(); time.sleep(REFRESH_SECONDS)` loop with a `time.ticks_ms()`-based poll scheduler and ship the three-view carousel. Introduce `clock_view.py` and `system_view.py` as blank stubs (per D-24). Install `Pin.irq(FALLING)` handlers on GP15 (KEY0) and GP17 (KEY1) with a shared software-debounce timestamp (D-13, D-14). Draw the three-dot page indicator at y=60 with 12 px spacing centered at x=52/64/76 (D-19, D-20, D-21). Refactor `weather_view.py` to cache its last-good weather tuple at module level so navigating back to Weather redraws instantly from cache (D-22 view-switch semantic).

After this plan the user experience is: boot -> Weather view with real weather (first fetch happens once at boot). Press KEY1 -> Clock view (blank content area + page dots). Press KEY1 -> System view (blank content area + page dots). Press KEY1 -> wraps back to Weather (redraws from cache instantly). KEY0 goes the other direction with the same wrap. A single physical press produces exactly one view change (debounce prevents chatter).

This plan does NOT implement the 600 s auto-refresh cadence, the boot "connecting..." / spinner, or the WEATHER-05 error-state polish that keeps the carousel navigable during error — those are Plan 03. What Plan 02 delivers is: navigation works, page dots update, stubs are reachable, boot goes to Weather.

Purpose: Deliver the user-visible carousel — the flagship Phase 2 capability. Six requirements land here (NAV-01 through NAV-06).

- `weather_view.py` — modified to cache and render-from-cache.
- `main.py` — heavily refactored: poll scheduler, IRQ install, carousel dispatch, page-dot draw pass.
