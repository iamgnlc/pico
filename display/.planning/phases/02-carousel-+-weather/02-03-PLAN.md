<objective>
Wire the 600 s Weather refresh cadence into the poll scheduler, formalize the boot-time "connecting..." + spinner sequence (D-23), and ensure the WEATHER-05 error state ("no wifi" / "no data") never blocks carousel navigation.

Concretely:
1. Add `_last_refresh_ms` timestamp and `_REFRESH_MS = 600_000` constant to `weather_view.py`. Expose `should_refresh(now_ms)` which returns True when `time.ticks_diff(now_ms, _last_refresh_ms) >= _REFRESH_MS`. `refresh(oled)` (already exists from Plan 02) updates `_last_refresh_ms` at the top so we don't re-fire on failure loops.
2. Add scheduler pass in `main.py`'s poll loop: after handling `_pending_dir`, call `if weather_view.should_refresh(now): weather_view.refresh(oled); ...redraw current view; oled.show()`. Refresh runs regardless of which view is currently displayed (per D-22 — cadence is decoupled from view-switch events).
3. Formalize the "connecting..." state (Plan 02 already renders this when `_cache_status == "pending"`) by ensuring `main.py` calls `weather_view.render(oled) + _draw_page_dots + oled.show()` BEFORE the initial `weather_view.refresh(oled)` call. This gives the user a visible "connecting..." screen while wifi.connect blocks.
4. Add an animated spinner drawn during the HTTP fetch phase in `weather_view.refresh(oled)`. Because MicroPython does not allow interrupting a blocking `urequests.get()` call, the spinner cannot animate DURING the single HTTP request. Instead, `refresh(oled)` draws two spinner frames spaced by a call to `oled.show()` in between: (a) one frame immediately after WiFi connects but BEFORE the `urequests.get()`, (b) a second frame is not possible during the same fetch. Planner interpretation of D-23's "visibly animating for at least one frame during the fetch phase" = the spinner is on-screen while the HTTP request is in flight. This satisfies the "visible during fetch" acceptance criterion with the minimum-complexity mechanism. If a future phase adds an async HTTP client, this can be upgraded to a true rotating animation.
5. WEATHER-05 verification: confirm that after Plan 02's carousel changes, error states ("no wifi"/"no data") render inside `weather_view.render(oled)`, and that `_draw_page_dots` still runs on top, so navigation continues to work during errors. No new code needed — Plan 02's cache-status model already handles this; this plan just verifies it explicitly in the operator check.

**Output:**
- `weather_view.py` (modified): scheduler predicate, refresh timestamp, spinner draw during fetch.
- `main.py` (modified): scheduler pass in the poll loop, connecting-state render before boot fetch.
