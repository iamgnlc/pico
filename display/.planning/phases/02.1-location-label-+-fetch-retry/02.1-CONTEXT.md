<decisions>
## Implementation Decisions

### Retry-cadence transition (60s fast-retry)
- **D-31:** The scheduler predicate `weather_view.should_refresh(now_ms)` inline-checks `_cache_status`:
  - `_cache_status == "ok"` → next refresh window is 600_000 ms
  - Any other status (`"pending"`, `"no_wifi"`, `"no_data"`) → next refresh window is 60_000 ms
  Two module-level constants: `_REFRESH_MS = 600_000` (already exists from Phase 2) and `_RETRY_MS = 60_000` (new). No boolean flag, no mutable interval variable — the existing `_cache_status` already carries all the signal needed.
- **D-32:** Cold-boot fetch failure is treated identically to any other failure — 60 s retry mode kicks in immediately. If the very first `weather_view.refresh(oled)` at boot leaves `_cache_status` in `"no_wifi"` or `"no_data"`, the poll scheduler's next `should_refresh(now_ms)` call fires True after 60 s. No special-case boot logic.
- **D-33:** Timestamp semantics unchanged from Phase 2: `_last_refresh_ms = time.ticks_ms()` is stamped at the START of `refresh(oled)` (existing pattern). A wifi-connect timeout takes ~20 s, so on failure the next `should_refresh` returns True approximately 40 s after `refresh` returns (60 s - 20 s wifi timeout). Acceptable — the retry cadence is measured from attempt start, not from attempt end. Simpler encapsulation, no caller-side timestamp management.

### Dropped decisions (retained here for historical audit)
- **D-26..D-30** — Location-label layout and data-source decisions. All dropped on 2026-07-18 with the Plan 02.1-01 revert. See git log for the reverted commits (`f93a8a4`, `564c0aa`, `9960ac6`) and the revert commit `b8823ab` for the audit trail.

<code_context>
## Existing Code Insights

### Reusable Assets
- **The `_cache_status` state machine** (from Phase 2 D-22) already encodes the failure/success signal that D-31's scheduler predicate reads. No new state field needed for retry-cadence transition.
- **The main.py scheduler branch** already reads `time.ticks_ms()` once per tick and calls `weather_view.should_refresh(now)`. Zero changes required in main.py — cadence transition is encapsulated inside `weather_view.should_refresh`.

### Established Patterns
- **Module-level UPPER_SNAKE_CASE constants for tunables** (`_REFRESH_MS`, `_POLL_MS`, `_DEBOUNCE_MS`, `_KEY0_PIN`, `_KEY1_PIN`). `_RETRY_MS = 60_000` follows this pattern.

### Integration Points
- **should_refresh interval switching** is a 2-line change inside the predicate — no impact on the caller. Existing callsite `weather_view.should_refresh(now)` stays byte-identical in main.py.
- **The 60 s retry mode's spinner behavior** is inherited from Phase 2's `db94c78 fix(02): suppress spinner during 'connecting...' (pending cache)` — the spinner still only fires on `_cache_status == "ok"` refreshes. During retry attempts (`no_wifi` or `no_data` cache states), no spinner. Consistent user experience.

</code_context>

<specifics>
## Specific Ideas

- **Cadence: 60 s from START.** No caller-side timestamp management. Matches Phase 2's existing `_last_refresh_ms = time.ticks_ms()` stamp discipline.
- **Cold-boot failure treated symmetrically.** No special-case boot logic. First failure at boot puts the device in 60 s retry mode immediately.

</specifics>

<deferred>
## Deferred Ideas

- **Distinguished error reasons** ("timeout" vs "no_connection" vs "api_error") — noted as an existing concern in `.planning/codebase/CONCERNS.md § "No Error Handling for WiFi/API Failures"`. WEATHER-09 satisfied by the coarse `_cache_status` signal without introducing new error taxonomies. Distinguished reasons remain deferred.
- **Retry-cadence tuning from device UI** — `_RETRY_MS` and `_REFRESH_MS` remain hard-coded module constants. A settings view would be a future addition and not in Phase 2.1 scope.

</deferred>
