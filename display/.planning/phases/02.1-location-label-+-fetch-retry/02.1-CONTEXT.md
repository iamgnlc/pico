<domain>
## Phase Boundary

Two additive Weather-view features:

1. **Location label (WEATHER-08).** Display the current city (or a regional fallback) somewhere in the Weather view alongside the existing icon (16, 16), temperature block, and drawn degree ring. The ip-api geolocation call inside `weather.current()` already returns location fields; Phase 2.1 exposes them and renders them.

2. **60-second fast-retry (WEATHER-09).** Extend the poll scheduler so a failed weather fetch triggers a 60 s retry cadence until the first successful fetch, then reverts to the 600 s default. The retry cadence must not block button responsiveness (existing IRQ + poll-loop pattern already handles this — Phase 2.1 only changes the interval selection, not the loop structure).

**Covers:** WEATHER-08, WEATHER-09 (2 requirements)

**Not in this phase:** Any change to navigation (NAV-01..06 already delivered by Phase 2), stub view content (Clock/System stay blank), the SH1107 driver, secrets/wifi/icons/text_render modules.

</domain>

<decisions>
## Implementation Decisions

### Layout — location label on the 128×64 canvas
- **D-26:** Location text is rendered **above** the temperature on the Weather view, with the temperature pushed slightly down so the combined location+temp block reads as vertically balanced with the icon on the left. Concrete reference coordinates (planner discretion for pixel-level tuning during on-device verification): location scale-1 centered at (88, 18); temp scale-2 centered at (88, 36) (moved from y=32 per Phase 2 fix `376615d`); degree ring `cy = 36 - 8 + 2 = 30`. Combined block spans y=14..43, safely inside reserved rows 0-53.
- **D-27:** Location text uses framebuf scale 1 (8 px tall). Truncation: slice the string to fit horizontally (max ~15 chars at 8 px per char centered on x=88 within the 128-px canvas) with no ellipsis — recognition of one's own city is high even with a few missing letters, and the alternative of eating 3 chars for "..." on a small display is worse.

### Location data source
- **D-28:** Primary field is ip-api's `city`. Fallback cascade when `city` is missing or empty: `regionName` → `countryCode`. At least one of the three will be populated in any successful ip-api response; the final displayed string is the first non-empty value in that order. If all three are empty (should not happen in practice), render nothing.

### weather.py contract change
- **D-29:** `weather.current()` extends its return contract from `(temp, code, is_day)` to `(temp, code, is_day, location)`. The `location` field is the resolved city/region/country string per D-28, or `None` on failure. Failure return becomes `(None, None, None, None)`. Callers (`weather_view.refresh`) update to unpack 4 values.
- **D-30:** Location is fetched on **every** refresh, alongside the weather data. No separate ip-api call — the location comes for free from the existing ip-api round-trip already inside `weather.current()`. `weather_view` gains a `_cached_location` module-level string, updated in lockstep with `_cached_temp` / `_cached_code` / `_cached_is_day`.

### Retry-cadence transition (60s fast-retry)
- **D-31:** The scheduler predicate `weather_view.should_refresh(now_ms)` inline-checks `_cache_status`:
  - `_cache_status == "ok"` → next refresh window is 600_000 ms
  - Any other status (`"pending"`, `"no_wifi"`, `"no_data"`) → next refresh window is 60_000 ms
  Two module-level constants: `_REFRESH_MS = 600_000` (already exists from Phase 2) and `_RETRY_MS = 60_000` (new). No boolean flag, no mutable interval variable — the existing `_cache_status` already carries all the signal needed.
- **D-32:** Cold-boot fetch failure is treated identically to any other failure — 60 s retry mode kicks in immediately. If the very first `weather_view.refresh(oled)` at boot leaves `_cache_status` in `"no_wifi"` or `"no_data"`, the poll scheduler's next `should_refresh(now_ms)` call fires True after 60 s. No special-case boot logic.
- **D-33:** Timestamp semantics unchanged from Phase 2: `_last_refresh_ms = time.ticks_ms()` is stamped at the START of `refresh(oled)` (existing pattern). A wifi-connect timeout takes ~20 s, so on failure the next `should_refresh` returns True approximately 40 s after `refresh` returns (60 s - 20 s wifi timeout). Acceptable — the retry cadence is measured from attempt start, not from attempt end. Simpler encapsulation, no caller-side timestamp management.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`weather.py:current()`** already fetches `http://ip-api.com/json/`. The response is available as a dict in local variable `loc` (lines 6-8); currently only `loc["lat"]` and `loc["lon"]` are extracted before `loc` goes out of scope. Adding a location extraction is a 3-4 line change in the try block using `loc.get("city")`, `loc.get("regionName")`, `loc.get("countryCode")` with the cascade per D-28.
- **`weather_view._center_text(oled, s, x_center, y_center, scale)`** already handles centered scaled text. Reusable for the location label (scale=1) without modification.
- **The `_cache_status` state machine** (from Phase 2 D-22) already encodes the failure/success signal that D-31's scheduler predicate reads. No new state field needed for retry-cadence transition.
- **The main.py scheduler branch** already reads `time.ticks_ms()` once per tick and calls `weather_view.should_refresh(now)`. Zero changes required in main.py — cadence transition is encapsulated inside `weather_view.should_refresh`.

### Established Patterns
- **Module-level UPPER_SNAKE_CASE constants for tunables** (`_REFRESH_MS`, `_POLL_MS`, `_DEBOUNCE_MS`, `_KEY0_PIN`, `_KEY1_PIN`). `_RETRY_MS = 60_000` follows this pattern.
- **`.format()` not f-strings** — mandatory for MicroPython idiom compliance.
- **Cache tuple + status enum** for stateful views (Phase 2 D-22). Extend by adding one more cached field, not by introducing a new state model.
- **Broad-exception fallback in weather.py** — the existing `except Exception: return None, None, None` becomes `return None, None, None, None`. The safety posture is unchanged; the tuple width grows by one.

### Integration Points
- **weather.py return contract change** ripples to exactly one caller: `weather_view.refresh(oled)`. Update the unpack to 4 values.
- **weather_view.py new state** (`_cached_location`, `_RETRY_MS`) is module-level; no external visibility. Neither `main.py`, `clock_view`, nor `system_view` needs to know.
- **should_refresh interval switching** is a 2-line change inside the predicate — no impact on the caller. Existing callsite `weather_view.should_refresh(now)` stays byte-identical in main.py.
- **The 60 s retry mode's spinner behavior** is inherited from Phase 2's `db94c78 fix(02): suppress spinner during 'connecting...' (pending cache)` — the spinner still only fires on `_cache_status == "ok"` refreshes. During retry attempts (`no_wifi` or `no_data` cache states), no spinner. Consistent user experience.
- **Layout impact on Phase 2's visual verification:** the temp y-anchor moves from 32 (Phase 2 baseline) to 36. Modest 4 px shift; still safely within rows 0-53. Operator visual re-check recommended on-device after Phase 2.1 execution.

<specifics>
## Specific Ideas

- **User's exact directive on layout:** "just above temp. push temp a bit down so they look vertically centered like the icon." Interpretation locked as D-26: location goes above temp; temp moves down modestly; combined block feels balanced against the icon on the left. Reference coords (88, 18) / (88, 36) are the planner's proposal — the operator has explicit permission to nudge during on-device verification if the visual balance doesn't match the intent.
- **Location string source cascade:** primary `city`; fallback cascade `city → regionName → countryCode`. The final displayed string is the first non-empty of the three. Do not concatenate — pick one.
- **Truncation:** slice to fit horizontally, no ellipsis. Max approximately 15 chars at scale 1 on a 128 px canvas when centered at x=88 (usable width around x=24..152 clamped to 128, so ~14-15 usable chars).
- **Cadence: 60 s from START.** No caller-side timestamp management. Matches Phase 2's existing `_last_refresh_ms = time.ticks_ms()` stamp discipline.
- **Cold-boot failure treated symmetrically.** No special-case boot logic. First failure at boot puts the device in 60 s retry mode immediately.

</specifics>

<deferred>
## Deferred Ideas

- **Portable device that changes locations** — Phase 2.1 fetches location every refresh, so location updates every 600 s (or 60 s during retries) automatically. Not deferred; delivered.
- **Distinguished error reasons** ("timeout" vs "no_connection" vs "api_error") — noted as an existing concern in `.planning/codebase/CONCERNS.md § "No Error Handling for WiFi/API Failures"`. WEATHER-09 satisfied by the coarse `_cache_status` signal without introducing new error taxonomies. Distinguished reasons remain deferred.
- **User-configurable location override** (WEATHER-07 v2 backlog) — skip ip-api geolocation, take a manually-configured location string. Separate v2 requirement; not in Phase 2.1 scope.
- **Multi-day / hourly forecast** (WEATHER-06 v2 backlog) — separate v2 requirement.
- **Retry-cadence tuning from device UI** — `_RETRY_MS` and `_REFRESH_MS` remain hard-coded module constants. A settings view would be a future addition and not in Phase 2.1 scope.
- **Location caching persistence across reboots** — location is memory-only (module state); a device reboot loses it and re-fetches on the next boot cycle. Persistence is out of scope; interacts with the "no persistence across reboots" architecture invariant from PROJECT.md.
- **Location display during pending/error states** — planner discretion. The natural choices are: (a) hide the location during `"pending"` (initial connecting) but show last-known during `"no_wifi"`/`"no_data"` if any prior fetch succeeded; (b) always render `_cached_location` if non-None, regardless of `_cache_status`. Recommend (b) as simpler — the render function checks `_cached_location is not None` before drawing, independent of the cache state.

</deferred>
