## Phase Goal

**As a** person using the Pico OLED when WiFi is flaky, **I want to** see the Weather view recover within ~60 seconds after a fetch failure instead of waiting 10 minutes, **so that** the device becomes accurate quickly once connectivity returns without me having to reboot.

<objective>
Deliver WEATHER-09: When a weather fetch fails, the device retries every 60 seconds until the first successful fetch, then reverts to the 600-second default cadence.

Purpose: Encapsulates the cadence transition entirely inside `weather_view.should_refresh(now_ms)` so `main.py`'s scheduler branch stays byte-identical. Uses the already-existing `_cache_status` signal (Phase 2 D-22) as the sole discriminator — no new boolean, no new mutable interval variable, no cadence-tracking bookkeeping on the caller.

Output: `weather_view.py` gains `_RETRY_MS = 60_000` module constant and an updated `should_refresh(now_ms)` predicate that returns True after `_REFRESH_MS` when `_cache_status == "ok"`, else after `_RETRY_MS`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| main.py scheduler → weather_view.should_refresh | Predicate reads module-level `_cache_status` (a str set only by `refresh(oled)` in the main-loop thread; no IRQ writer). Read-only from the scheduler's perspective. |
| weather_view._cache_status writes | `refresh(oled)` is the sole writer of `_cache_status`. IRQ handlers (`_on_key0`, `_on_key1` in main.py) never touch `_cache_status`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-02-01 | Denial of Service | `should_refresh` firing every poll tick when `_cache_status != "ok"` | mitigate | `_last_refresh_ms` is stamped at the START of `refresh(oled)` (D-33 invariant, unchanged). A failed fetch consumes ~20s of wifi timeout, then `_last_refresh_ms` remains stamped from attempt start — the next `should_refresh` fires True only after the full 60s window elapses from that start-of-attempt timestamp. No tight loop. |
| T-02.1-02-02 | Tampering | Race between `refresh` writing `_cache_status` and `should_refresh` reading it | accept | Both callers run on the main loop thread (Pico W Cortex-M0+ single-threaded userland; IRQ handlers only write `_pending_dir`, not `_cache_status`). No cross-thread access. Reads of a Python str reference are atomic in MicroPython. |
| T-02.1-02-03 | Denial of Service | `_RETRY_MS` set too low (e.g. 1000) could hammer the API | mitigate | Value is a module-level constant `60_000` (60 seconds). Not user-configurable at runtime; requires a source edit + file copy to change. ip-api free tier tolerates 60s polling well under any documented rate limit. |
| T-02.1-02-04 | Information Disclosure | Longer visibility of "no wifi" / "no data" text during 60s retry cycles | accept | These are the intentional error states from Phase 2 (WEATHER-05). Displaying them for longer during retries is the desired UX — the user needs to see the device is offline. No new information exposed. |
| T-02.1-02-SC | Tampering | npm/pip/cargo installs | mitigate | N/A — this plan adds no new packages. Only the existing `time` module (MicroPython stdlib) is used; no installer touchpoint. |

Zero HIGH severity threats. `security_asvs_level: 1`, `security_block_on: high`: standard visual verify checkpoint is the plan's gate.
