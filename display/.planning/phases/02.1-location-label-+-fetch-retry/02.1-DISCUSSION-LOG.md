## Location label layout

### Q1: Position on 128×64 canvas

| Option | Description | Selected |
|--------|-------------|----------|
| Bottom of content area, y~48 | Location centered at y=48, between temp and dot strip. | |
| Top of screen, y~4 | Location as a header at y=4; pushes icon and temp down. | |
| Right of temp, same row | Scale-1 text right of degree ring; almost every city name would need truncation. | |
| **User custom: "just above temp. push temp a bit down so they look vertically centered like the icon"** | Location above temp; temp moves down; combined block visually balanced with the icon on the left. | ✓ |

**Rationale:** Neither predefined option matched the user's mental model. Interpretation locked as D-26 with reference coords (loc center (88, 18), temp center (88, 36), ring cy=30). Planner-discretion permitted to nudge coordinates during on-device visual verification.

### Q2: Font scale

| Option | Description | Selected |
|--------|-------------|----------|
| Scale 1 (8px tall) | Default framebuf font. ~15 chars fit at 128px width. Cheapest to draw. | ✓ |
| Scale 2 (16px tall) | Larger, would compete with temp for visual weight; truncates most city names. | |

**Rationale:** Location is a secondary label; scale-1 keeps hierarchy correct (temp is the primary reading).

### Q3: Truncation strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Truncate to fit, no ellipsis | Slice to N chars, render as-is. Preserves as many characters of the actual name as possible. | ✓ |
| Truncate + ellipsis "..." | Eats 3 chars for the indicator. On a 128px canvas that's 24px of city name lost. | |
| No truncation, let overflow clip | Framebuf clips at x=128. Visually broken for long names. | |

**Rationale:** Users generally know their own city name; partial rendering is still recognizable, and 3 chars is a meaningful fraction of a 15-char budget.

## Location data source

### Q1: Which ip-api field(s)

| Option | Description | Selected |
|--------|-------------|----------|
| `city` only | Most recognizable, shortest label. | ✓ |
| `city, countryCode` (e.g. "London,GB") | Disambiguation for common city names; adds 3 chars overhead. | |
| `regionName` only (e.g. "California") | Always populated when ip-api resolves; less precise than city. | |

**Rationale:** City is the natural choice for a personal device. Country disambiguation isn't needed for a fixed-location deployment.

### Q2: Fallback strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Fall back to regionName, then countryCode | Cascade covers datacenter-IP / VPN edge cases where city may be null. | ✓ |
| Fall back to "?" or placeholder | Simpler; less informative. | |
| Hide the location label entirely | Preserves layout slot; complicates render function with conditional anchor logic. | |

**Rationale:** Cascade is a 3-line change (`city or regionName or countryCode` with empty-string handling). Always displays something when ip-api resolves; renders nothing only in the exceedingly rare case of all three being empty.

## Location cache model + weather.py contract

### Q1: How does location get into weather_view's cache?

| Option | Description | Selected |
|--------|-------------|----------|
| Extend `weather.current()` tuple to 4-value | Smallest surface change; single API call already fetches ip-api and open-meteo. | ✓ |
| Separate `weather.location()` function | Adds a distinct call; would double ip-api hits unless careful. | |
| Fetch location once at boot, cache indefinitely | Reduces API load; portable device wouldn't update location until reboot. | |

**Rationale:** Location comes for free with the existing ip-api round-trip. Extending the tuple is the minimum-change path.

### Q2: Cadence — fetch every refresh or only on failure recovery

| Option | Description | Selected |
|--------|-------------|----------|
| Every refresh, alongside weather | Zero-cost since ip-api is already on the same round trip. | ✓ |
| Only on failure recovery | Preserve last-known across refreshes; complicates state machine for no benefit given the ip-api call happens anyway. | |

**Rationale:** No network-cost savings from the alternative because the ip-api call happens regardless.

## Retry-cadence transition model

### Q1: Interval selection mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| `should_refresh(now_ms)` reads `_cache_status` inline | 60000 ms if not "ok", 600000 ms if "ok". Zero new state. | ✓ |
| Separate `_retry_mode` boolean flag | Explicit but redundant with `_cache_status`. | |
| Mutable `_current_refresh_ms` variable | Simplest predicate; muddles constants-vs-state distinction. | |

**Rationale:** `_cache_status` already carries the failure/success signal from Phase 2. Reading it in the predicate reuses that state with zero additions.

### Q2: Cold-boot fetch failure behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Any failure = 60s retry, including boot | Symmetric; matches user-facing spec "retry every 1 min until connected". | ✓ |
| 60s retry only after first success | Weird asymmetry; user waits 10 min on cold-boot wifi failure before retries kick in. | |

**Rationale:** No special-case boot logic. First failure → 60s mode immediately.

### Q3: Timestamp semantics — 60s from START or END?

| Option | Description | Selected |
|--------|-------------|----------|
| 60s from START | Matches Phase 2's existing `_last_refresh_ms = time.ticks_ms()` stamp discipline. | ✓ |
| 60s from END | Guarantees 60s "quiet time" between attempts; requires caller-side timestamp management. | |

**Rationale:** Encapsulation preserved. Actual quiet time between attempts on a failed wifi-connect timeout is ~40s (60s - 20s wifi timeout), which is acceptable for the "retry every 1 min" spec.

## Claude's Discretion

- Reference coordinates for D-26 layout: location center (88, 18), temp center (88, 36), ring cy=30. Planner-discretion to nudge during on-device visual verification. User's explicit permission granted via the "look vertically centered like the icon" directive.
- Exact truncation formula: max chars = `min(len(location), width_budget // 8)`. Simple string slice.
- Fallback cascade implementation: `city or regionName or countryCode or None` (Python evaluates empty strings as falsy).
- Location display when `_cached_location is None` (e.g. before first successful fetch, or all three fields empty): draw nothing at the location y-anchor. Temp remains where it is; no vertical compensation. This is the simplest render logic.
