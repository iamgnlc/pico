## Overview

Four vertical phases + one inserted decimal phase take the existing single-view weather demo to a fully navigable three-view carousel. Phase 1 closes the credential leak and fixes the degree-symbol rendering. Phase 2 builds the complete carousel shell and delivers the polished Weather view. Phase 2.1 was inserted mid-milestone for the WEATHER-09 retry logic after Phase 2 shipped. Phases 3 and 4 replace the Clock and System stubs with real implementations. A post-verify refactor (quick task `260718-rename-bootstrap`) consolidated `wifi.py` + `weather.py` into `bootstrap.py` for cleaner semantics before the milestone tag.

## Milestone Summary

### Decimal Phases

- **Phase 2.1: Fetch Retry (INSERTED)** — added after Phase 2 shipped, when the operator flagged that a failed weather fetch would leave the device stuck for the full 10-minute cadence before retrying. Originally scoped as "Location Label + Fetch Retry"; the location-label plan (02.1-01) was reverted after human-verify caught a layout regression, and the phase narrowed to retry-only. Delivered WEATHER-09 (60s retry until success, then revert to 10-min).

### Key Decisions

**Phase 2 (D-13..D-25):**
- Carousel-only navigation (no menu/list)
- Poll-scheduler with `_POLL_MS = 100`; IRQ handlers set `_pending_dir` for main-loop dispatch
- Page dots at y=60; view content in rows 0-53
- On-view-switch = instant redraw of cached data
- D-23 (spinner during weather HTTP fetch) — later RETIRED 2026-07-18

**Phase 2.1 (D-31..D-33):**
- `should_refresh` predicate reads `_cache_status` inline (600s if `"ok"`, else 60s)
- Cold-boot fetch failure symmetrically triggers retry mode
- `_last_refresh_ms` stamped at START of `refresh(oled)` (stamp-at-start invariant)
- D-26..D-30 (location label + weather 4-tuple) — RETIRED with the 02.1-01 revert

**Phase 3 (D-34..D-40, D-43-bis):**
- Clock format = `HH:MM` 24h no seconds (minute-boundary repaints)
- NTP re-sync every 1h (later relaxed to 6h via quick task)
- 60s retry until first success (mirrors Phase 2.1 D-31)
- WiFi drop post-first-sync keeps time visible; `--:--` means "never synced"
- Two pure predicates + one sync fn; main.py gates re-render with `_current_idx == 1`
- Single-boolean `_synced` state (no enum)
- CLOCK-02 REDIRECTED mid-phase: from hardcoded `TZ_OFFSET` to auto-derived from ip-api + persisted to `tz_offset.txt`

**Phase 4 (D-41..D-45):**
- View-switch-only refresh (no `should_tick` predicate)
- Signal strength = 4 drawn bars (no dBm text); RSSI thresholds -55/-65/-75
- Layout = 3-line left-aligned list (SSID / IP / Signal + bars) at scale 1
- IP field = WAN IP (piggybacked on ip-api `query` field); dual-gate render
- Offline UX = uniform `--` for all three fields
- System view is READ-ONLY (no reconnect attempts)

**Cross-phase / post-verify (2026-07-18):**
- **Quick 260718-scope-cleanup-phase-2.1**: dropped WEATHER-08 from v1 scope after Plan 02.1-01 revert
- **Quick 260718-remove-spinner-tune-cadence**: retired D-23 (spinner) + updated D-35 NTP cadence 1h → 6h
- **Quick 260718-rename-bootstrap**: renamed `weather.py` → `bootstrap.py`, absorbed `wifi.py` into `bootstrap._wifi_connect`, consolidated to single 6-tuple `fetch()` return
